# デザイン仕様 (DESIGNS.md)

参加者をチームにランダム振り分けする「くじ引き」Webサービスの **デザイン仕様** を定義する。

このドキュメントは以下を目的とする。

- 現行デザイン（最低限のスタイル）の網羅的な記録 = 刷新前のベースライン
- デザイン刷新時に「維持すべき体験/情報設計」と「自由に変えてよい見た目」の切り分け
- 機能（[FUNCTIONS.md](./FUNCTIONS.md)）と独立して見た目を作り替えるための参照仕様

> 注: 現行は Vanilla JS で DOM を文字列生成し、単一の `public/styles.css` で素朴にスタイリングしている。刷新ではデザインシステム（トークン化）導入を前提に整理する。

---

## 1. デザイン目標

| 観点 | 現行 | 刷新方針 |
|------|------|----------|
| トーン | 機能優先・最低限 | 「くじ引き」のワクワク感を演出しつつ会場で映える、洗練された見栄え |
| デバイス | スマホ前提（max-width 480px）+ 会場表示（720px） | モバイルファースト維持 + 大画面の視認性強化 |
| 一貫性 | アドホックなクラス | デザイントークン化（色/余白/角丸/影/タイポ/モーション） |
| 視認性 | 未検証（小さめ文字・低コントラスト箇所あり） | **会場の遠距離・スマホ屋外を想定した高視認性を必須要件化**（§10） |
| アクセシビリティ | 最低限（コントラスト未検証） | WCAG AA コントラスト、フォーカス可視、動き低減対応 |
| ダークモード | なし | **必須対応**。OS設定追従 + 手動切替（§8） |
| 国際化 (i18n) | 日本語ハードコード・佐賀弁固定・青系固定 | **必須対応**。文言の外部化・言語切替・RTL 考慮・テーマ差し替え（§9） |

## 4. アニメーション・演出仕様（体験の核 = 維持必須）

「くじ引き」の楽しさを担う要素。刷新でも **同等以上の体験** を維持する。

| 演出 | 現行実装 | 維持すべき意図 |
|------|----------|----------------|
| 抽選スロット | 80ms ごとにチーム名を切替、1.2〜2.0秒間 | 「抽選している」緊張感・期待感 |
| 抽選中ドット | `抽選中...` のドットアニメ（@keyframes dots） | 処理中フィードバック |
| 決定演出 | `pop`（scale 0.8→1.05→1）+「決定！」ラベル | 結果確定の高揚 |
| 紙吹雪風 | `::before/::after` の浮遊円（@keyframes float） | 祝祭感 |
| 触覚 | `navigator.vibrate(30)` 等 | モバイルでの実感 |
| トースト | 下からスライドイン（transform + opacity） | 操作成功の通知 |
| 会場タイトル脈動 | `pulse`（opacity 1↔0.7, 2s） | 大画面でのライブ感 |

> 刷新メモ:
> - `prefers-reduced-motion` に対応し、動き低減設定時は演出を抑制する（現行は未対応）。
> - 紙吹雪は2点のみで簡素。刷新では本格的な confetti（canvas等）への差し替えも可。ただし会場ディスプレイの負荷に留意。

## 5. 刷新で導入するデザインシステム（提案）

### 5.1 デザイントークン（CSS Custom Properties 例）

役割ベースのトークンを単一の出所（source of truth）とし、**ライト/ダーク両モード・テーマ差し替え・i18n** をすべてトークン上書きで成立させる。

> **配色値の扱い**: 本書は **トークンの役割名・モード構造・制約**（コントラスト等）を規定する。
> 具体的な配色値（hex・グラデーション）は**デザインツールで確定**する想定で、以下のコード内の色値は**暫定の参考値**。
> ツールの出力は §7・§10 のコントラスト要件と §8 のライト/ダーク両対応を満たすこと。
> 一方で **トークン名・階層・モード切替の仕組みは固定の契約**（SPA がこれに依存）であり、値だけ差し替える前提で残す。

```css
:root {
  /* === Color roles (light = default) ===
     ↓ 色値はすべて暫定（デザインツールで確定）。役割名・構造は固定。 */
  --color-bg: #f4f6fb;
  --color-bg-elevated: #ffffff;     /* カード等の浮いた面 */
  --color-surface: #ffffff;
  --color-surface-muted: #f1f5f9;   /* チームカード等の控えめな面 */
  --color-border: #e2e8f0;
  --color-text: #0f172a;            /* 本文（AA 確保のため濃いめ） */
  --color-text-muted: #475569;      /* 補助テキストも AA を満たす濃さに引き上げ */
  --color-text-on-accent: #ffffff;  /* アクセント面上の文字 */

  --color-primary: #2563eb;
  --color-primary-strong: #1d4ed8;
  --color-primary-soft: #dbeafe;    /* 淡い背景面 */
  --color-accent: var(--color-primary); /* テーマで差し替え */
  --color-success: #047857;         /* AA 確保のためやや濃く */
  --color-danger: #dc2626;
  --color-warning: #b45309;

  /* === Radius === */
  --radius-sm: 8px;
  --radius-md: 12px;
  --radius-lg: 20px;
  --radius-pill: 999px;

  /* === Elevation (影の段階) === */
  --shadow-1: 0 1px 2px rgba(15,23,42,.06), 0 1px 3px rgba(15,23,42,.08);
  --shadow-2: 0 4px 12px rgba(15,23,42,.10);
  --shadow-3: 0 12px 32px rgba(15,23,42,.16);
  --shadow-focus: 0 0 0 3px color-mix(in srgb, var(--color-accent) 45%, transparent);

  /* === Gradient (祝祭感の演出に流用) === */
  --gradient-festive: linear-gradient(135deg, #eff6ff 0%, #dbeafe 100%);

  /* === Spacing scale (4px グリッド) === */
  --space-1: 4px; --space-2: 8px; --space-3: 12px;
  --space-4: 16px; --space-5: 20px; --space-6: 24px;
  --space-8: 32px; --space-12: 48px;

  /* === Typography === */
  --font-base: system-ui, "Hiragino Sans", "Hiragino Kaku Gothic ProN",
               "Noto Sans JP", "Noto Sans", sans-serif; /* i18n: 多言語フォールバック */
  --font-display: var(--font-base); /* 会場表示用に差し替え可 */
  --leading-tight: 1.25;
  --leading-normal: 1.6;            /* 多言語の行間確保 */

  /* Type scale (clamp で画面幅に追従 = 視認性) */
  --fs-caption: 0.85rem;
  --fs-body: 1rem;
  --fs-h2: 1.25rem;
  --fs-h1: clamp(1.5rem, 4vw, 2rem);
  --fs-result: clamp(2rem, 8vw, 3.5rem);   /* くじ結果 */
  --fs-slot: clamp(2.25rem, 9vw, 4rem);    /* 抽選スロット */

  /* === Layout === */
  --container-w: 480px;
  --container-w-display: 1024px; /* 大画面の視認性向上のため拡大 */
  --tap-target-min: 44px;        /* タップ領域の最小サイズ */

  /* === Motion (reduced-motion でゼロ化する起点) === */
  --motion-fast: 150ms;
  --motion-base: 250ms;
  --motion-slot: 1600ms;
  --ease-out: cubic-bezier(.16, 1, .3, 1);
}

/* OS 設定がダークのときの上書き（手動切替は §8 の data-theme で上書き） */
@media (prefers-color-scheme: dark) {
  :root:not([data-theme="light"]) {
    --color-bg: #0b1120;
    --color-bg-elevated: #131c2e;
    --color-surface: #131c2e;
    --color-surface-muted: #1c2740;
    --color-border: #283448;
    --color-text: #f1f5f9;
    --color-text-muted: #b6c2d4;     /* ダークでも AA を満たす明るさ */
    --color-primary: #60a5fa;        /* 暗背景で沈まないよう明度を上げる */
    --color-primary-strong: #93c5fd;
    --color-primary-soft: #1e3a5f;
    --color-success: #34d399;
    --color-danger: #f87171;
    --color-warning: #fbbf24;
    --gradient-festive: linear-gradient(135deg, #1e293b 0%, #1e3a5f 100%);
    --shadow-1: 0 1px 2px rgba(0,0,0,.4);
    --shadow-2: 0 4px 12px rgba(0,0,0,.5);
    --shadow-3: 0 12px 32px rgba(0,0,0,.6);
  }
}

/* 動き低減（演出の起点トークンを無効化） */
@media (prefers-reduced-motion: reduce) {
  :root {
    --motion-fast: 0ms; --motion-base: 0ms; --motion-slot: 0ms;
  }
}
```

- 役割ベースの命名により、**テーマ（地域・イベント色）・ライト/ダーク・言語フォント** の差し替えをトークン上書きのみで実現する。
- 補助テキスト色・成功色は現行より濃くし、コントラスト AA を満たす（§7・§10）。
- フォントスタック・タイプスケールは `clamp()` で画面幅に追従させ、スマホ〜会場大画面まで視認性を担保する。

### 5.2 コンポーネント分解（再利用単位）

| `Card` | 情報の箱 | `.card` |

### 5.3 ビジュアル品質の引き上げ（見栄え）

「最低限」から脱却するための具体方針。すべて §5.1 のトークンで実現する。

| 要素 | 現行 | 刷新方針 |
|------|------|----------|
| 面の階層 | 影は1段階のみ（`shadow-card`） | `--shadow-1〜3` のエレベーション階層で奥行きを表現 |
| カード | フラットな白箱 | 角丸を `--radius-lg` に、`--color-border` の繊細な枠線 + ホバーで持ち上げ |
| プライマリボタン | 単色塗り | グラデ/微光沢、押下で沈むマイクロインタラクション、`--tap-target-min` 確保 |
| くじボタン | 通常ボタン | ヒーロー化（大型・脈動の誘目）、引いた瞬間の触感的フィードバック |
| 結果カード | 静的グラデ | 紙吹雪 + 拡大 + アクセントグラデで「ハレ」の演出（§4 維持） |
| チームカード | 左ボーダー1色 | チームごとに色相を割り当て、色＋ラベルで識別（色覚多様性に配慮し色のみに依存しない） |
| 進捗表示 | テキストのみ | 進捗バー/リング化で残り枠を直感的に |
| アイコン | なし | 軽量なラインアイコンで意味を補助（文字依存を緩和、i18n にも有利） |
| 余白 | ばらつき | 4px グリッド（`--space-*`）で律する |

> チーム色は「色だけで意味を伝えない」原則を守る（必ずチーム名ラベルを併記）。色覚多様性・ダークモード両対応のため、彩度・明度はトークンで管理する。

## 6. 画面別デザイン要件

### 6.1 参加者画面

- **情報の優先順位**: ①くじを引く操作 ②自分の結果 ③全体進捗 ④チーム一覧。
- 必須要素: タイトル、event_code、進捗（参加/総枠・残り枠）、表示名入力、くじボタン、結果カード、チーム一覧、display/admin への導線。
- 体験要件: 抽選演出（§4）、結果は大きく祝祭的に。再訪時は既存結果を即表示。
- 刷新案: ヒーロー的なくじボタン、結果の共有性向上、残り枠の視覚的バー化。

### 6.2 管理者画面

- 情報密度が高い。現状はカードの縦積み。
- 必須要素: 現在のチーム名一覧、チーム名再生成（割当0人時のみ活性）、イベント名、チーム数/総枠（均等配分の自動反映）、プリセット/カスタム構成、設定保存、トークン入力、割当リセット/全リセット（二段階確認）、進捗ステータス。
- フィードバック: インラインのエラー/成功メッセージ + トースト。
- 刷新案: セクション見出しの明確化、危険操作（全リセット）の視覚的隔離、トークン保持状態の可視化、フォームのバリデーション表示改善。

### 6.3 会場表示画面

- 大画面・遠距離視認前提。`max-width 720px`、文字大きめ。
- 必須要素: タイトル（脈動）、進捗、参加用QR + URL、チーム一覧（メンバー表示）。
- 刷新案: コントラスト最大化、チームカラー分け、QRと残り枠の視認性強化、自動更新時のちらつき防止（差分更新/トランジション）。

### 6.4 共通（ローディング/エラー）

- ローディング: 中央の「読み込み中」。刷新ではスケルトン or スピナー。
- エラー: 赤メッセージ + トップ導線。刷新では再試行ボタンを検討。

## 7. アクセシビリティ要件（刷新で達成）

現行は未検証。刷新で以下を満たす。

- コントラスト比: テキスト/背景で WCAG AA（通常4.5:1、大文字3:1）以上。
- フォーカス可視: キーボード操作で focus ring を明示。
- フォーム: `label` と入力の関連付け（現行は概ね対応）、エラーは `aria-live` で通知。
- 動き低減: `prefers-reduced-motion` で演出（スロット/紙吹雪/脈動/トースト）を抑制。
- モーダル: フォーカストラップ、`Esc` で閉じる、`role="dialog"`/`aria-modal`。
- 触覚/音は補助手段（必須にしない）。

## 8. ダークモード対応（必須）

### 8.1 方針

- **OS 設定追従を既定**とし、`@media (prefers-color-scheme: dark)` でトークンを上書き（§5.1）。
- **手動切替**を併設: `<html data-theme="light|dark">` を最優先とし、未指定時のみ OS 設定に従う。
  - 切替UIは「自動 / ライト / ダーク」の3状態。選択は `localStorage` に保持。
  - 初期描画前にテーマを確定させ、画面のちらつき（FOUC）を防ぐ（インラインの先頭スクリプトで `data-theme` を適用）。
- 色は §5.1 のダークトークンに集約し、コンポーネント側でハードコードしない。

### 8.2 ダーク時の品質要件

- 純黒（#000）を避け、`--color-bg`（濃紺寄り）で目の負担を軽減。
- 暗背景で沈むアクセント色は明度を上げる（例: primary を `#60a5fa` 系へ）。
- 影は弱くなるため、`--color-border` と面の明度差で階層を表現。
- ダークでも本文・補助テキストとも WCAG AA を満たす（§10 で検証）。
- 画像/QR: QR は暗背景で読めるよう **白パディング付きの白地で表示**（反転させない）。

### 8.3 会場表示モード

- 会場の照明環境に応じてライト/ダークを選べるようにする（プロジェクター投影はダーク、白基調の会場はライトが見やすい等）。

## 9. 国際化（i18n）対応（必須）

現行は日本語をハードコードし、チーム名語彙も佐賀弁固定。刷新では文言とデータを分離する。

### 9.1 文言の外部化

- UI 文言をロケール別の辞書（例: `locales/ja.json`, `locales/en.json`）へ外出しし、キー参照に置き換える。
- 既定ロケール `ja`。言語判定は「URL/設定 > `localStorage` > `navigator.language` > 既定」の順。
- 日付・数値はロケール対応 API（`Intl.DateTimeFormat` / `Intl.NumberFormat`）で整形。
- 文言は結合（concatenation）を避け、プレースホルダ付きテンプレートで管理（語順が言語で変わるため）。
  - 例: `"{name} さんは {team} に決定！"` ↔ `"{name}, you're on {team}!"`
- 複数形は ICU MessageFormat 等のルールに委ねる（英語の単複など）。

### 9.2 レイアウトの言語耐性

- 文字列長の伸縮（独語など長くなる言語）に耐えるよう、固定幅・1行前提を避け折り返し可能に。
- フォントスタックに多言語フォント（Noto 系）をフォールバックとして含める（§5.1）。
- **RTL（右から左）考慮**: 方向依存の指定は論理プロパティ（`margin-inline-start` 等）を使い、`dir="rtl"` で鏡像化できる土台にする（初期対応言語に RTL が無くても将来に備える）。

### 9.3 コンテンツ（データ）側との分担

- **チーム名語彙はデザインでなくデータ/機能側の関心事**（FUNCTIONS.md §5.6）。地域・テーマごとに差し替え可能とし、佐賀弁は「既定テーマの一例」と位置づける。
- イベントタイトル等のユーザー入力はそのまま表示（翻訳対象外）。XSS エスケープは維持（FUNCTIONS.md §6）。

### 9.4 注意

- UI 言語（i18n）とイベントのチーム名テーマは独立。英語UIで佐賀弁チーム名、なども成立させる。
- `<html lang>` を選択ロケールに同期（スクリーンリーダー・フォント選択のため）。

## 10. 視認性・レジビリティ強化（必須）

会場の遠距離視認・スマホ屋外利用を想定し、「読めること」を最優先要件とする。

| 観点 | 要件 |
|------|------|
| コントラスト | 本文・補助テキストとも WCAG AA（通常4.5:1 / 大文字3:1）以上。ライト/ダーク双方で検証 |
| フォントサイズ | `clamp()` で画面幅に追従（§5.1）。会場表示は特に大きく |
| 行間・字間 | 多言語で詰まらないよう `--leading-normal` を確保 |
| タップ領域 | 操作要素は `--tap-target-min`（44px）以上 |
| 情報の冗長化 | 色のみに依存せず、ラベル/アイコン/形でも区別（チーム識別・状態表示） |
| 進捗の可読化 | 残り枠・参加状況を数値＋バー/リングで二重に提示 |
| ちらつき防止 | ポーリング更新時は全置換でなく差分更新＋短いトランジションで視線を奪わない |
| 屋外・低照度 | ダークモード＋十分なコントラストで明所/暗所どちらも読める |
| 会場大画面 | `--container-w-display` 拡大、メンバー名が遠くから読めるサイズに |

## 11. 移植・テーマ運用との関係

- デザインは機能ブランチ（FUNCTIONS.md §7.1 の `deploy/*`）と独立して刷新可能。SPA は API 仕様にのみ依存するため、見た目の変更が業務ロジックに波及しない。
- テーマ（色・チーム名語彙・タイトル）はイベント/地域ごとに差し替え可能にする方針。語彙は機能側の定数（FUNCTIONS.md §5.6）、見た目はトークン（§5.1）で制御する。
- 外部依存の QR 生成（FUNCTIONS.md §6 メモ）は、デザイン面でもオフライン/印刷品質の観点からクライアント生成への移行を推奨。
